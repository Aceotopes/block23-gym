# ADR-004 — Payment.paymentMethod Required with CASH Default

**Status:** Accepted
**Date:** 2026-06-20
**Phase:** Phase 1 — Schema Redesign

---

## Context

The developer confirmed that all payment transactions must record whether they were collected as cash, GCash, or PayMaya. This enables daily cash reconciliation, cash vs. digital revenue reporting, and future payment gateway integration.

The challenge: the existing `Payment` table contains rows with no payment method data (the field did not exist in V1). A migration adding a NOT NULL column to a table with existing rows requires either a default value or a nullable column.

---

## Decision

Add `paymentMethod PaymentMethod @default(CASH)` to the `Payment` model.

- The field is NOT NULL — every payment record always has a payment method
- The Prisma migration applies `CASH` to all pre-existing payment rows
- Going forward, all payment creation flows require an explicit `paymentMethod` argument — no default is used in application code, only in the migration

### Rationale for the CASH default on existing rows

The gym is in its early operational stage. All payments recorded before Phase 1 were processed manually by staff in person. Given the context (Philippines, small gym, no digital payment infrastructure in place yet), cash payments are the overwhelmingly likely method for all historical records.

This assumption is documented here and in the seed file comments. If the assumption is wrong for specific records, those records can be corrected via the ADMIN payment edit feature (Phase 3).

### PaymentMethod enum

```
PaymentMethod
  CASH
  GCASH
  PAYMAYA
```

No `OTHER` value is included. The developer confirmed these three methods are the complete set for the foreseeable future. If a new method is needed, it will be added to the enum via migration at that time.

---

## Consequences

**Positive:**
- `paymentMethod` is never null in application code — no null-handling branches needed in filters, reports, or display components
- Daily cash reconciliation is possible: sum payments where `paymentMethod = CASH`
- Revenue breakdown by method (cash vs. digital) is a simple GROUP BY query
- When payment gateway integration is added (future), the field already exists — only the creation logic changes

**Negative:**
- Historical payment records will show `CASH` regardless of actual method. For an early-stage gym with 1–2 staff and all-in-person transactions, this is an acceptable and documented approximation.
- If the gym was processing GCash or PayMaya payments before Phase 1 (unlikely given the system's current state), those records will be incorrectly labeled as CASH. They can be corrected by ADMIN via Phase 3's payment edit feature.

---

## Alternatives Considered

### Nullable `paymentMethod PaymentMethod?`
Existing rows get `paymentMethod = null`. No data assumption is made. Rejected: every UI component displaying payment method, every report filtering by method, and every summary query must handle null as a third state alongside the enum values. This permanent complexity is not worth the precision of admitting uncertainty about old records.

### Free-text `paymentMethod String?`
Maximum flexibility for payment method values. Rejected: inconsistent strings (e.g., "gcash" vs. "GCash" vs. "GCash QR") make reliable filtering impossible and require UI normalization code.

### Separate `PaymentMethod` model (lookup table)
Admin-configurable payment methods as a database table. Rejected: payment methods are regulatory/infrastructure-bound (cash, GCash, PayMaya), not business-configurable. An enum is the appropriate type for a bounded, stable set of values.
