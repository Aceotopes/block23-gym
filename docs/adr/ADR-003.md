# ADR-003 — Partial Audit Log with AuditAction Enum

**Status:** Accepted
**Date:** 2026-06-20
**Phase:** Phase 1 — Schema Redesign

---

## Context

The developer confirmed a partial audit log requirement: log client deletions and payment changes only. Full audit logging (every create, update, delete on every model) was explicitly out of scope.

Two design questions were open:

1. Should `action` be a free-text `String` or a typed enum?
2. What should happen to audit log entries if the staff user who performed the action is later removed from the system?

---

## Decision

### `AuditAction` enum

```
AuditAction
  DELETE_CLIENT
  EDIT_PAYMENT
```

The action field uses an enum rather than a free-text string. The scope of audit logging is narrow and well-defined. An enum provides:
- Compile-time type safety — typos are caught before deployment
- Reliable query filtering — `WHERE action = 'DELETE_CLIENT'` is exact, not a string match
- A clear extension point — adding a new action type is a deliberate, documented schema change

### `beforeState Json?`

The snapshot field is named `beforeState` (not `metadata`) to make clear it holds the state of the record immediately before the change was made. This field is optional — for events where a before-state is not meaningful, it can be null.

### `AuditLog` is immutable

No `updatedAt` field. No UPDATE or DELETE operations are permitted on `AuditLog` records from application code.

### `onDelete: SetNull` on `userId`

When a staff account is removed from the system, existing audit log entries retain all their data but `userId` is set to null. The UI displays these as "Deleted User." The audit trail is preserved; only the user attribution is lost.

This was chosen over `Restrict` (blocking user deletion while audit logs reference them) because staff accounts may need to be removed even when they have a long audit history.

---

## Consequences

**Positive:**
- Destructive and financial operations are traceable to a specific staff member and timestamp
- `beforeState` enables "what did this look like before the edit?" queries without consulting git history or backups
- The narrow scope (two action types) keeps the audit table small and queries fast
- `AuditAction` enum prevents silent data inconsistencies from free-text action strings

**Negative:**
- Adding a new action type (e.g., if EDIT_CLIENT is later needed) requires a schema migration. This is intentional — it forces the decision to be deliberate.
- `userId` becomes null when a user is deleted. If many staff accounts are cycled over time, a growing percentage of audit entries will show "Deleted User." This is acceptable for a small team (1–2 staff).

**Where audit entries are written:**
- `DELETE_CLIENT` — written inside `deleteClient()` server action (Phase 1 code change)
- `EDIT_PAYMENT` — written inside the payment edit server action (Phase 3)

---

## Alternatives Considered

### Free-text `action String`
Maximum flexibility — new action types need no migration. Rejected: inconsistent string values across action files (e.g., "delete_client" vs. "deleteClient" vs. "DELETE_CLIENT") would silently corrupt filtering. The narrow scope makes an enum strictly better.

### Full audit log (all models, all operations)
Log every create, update, and delete across all models. Rejected by the developer — out of scope for current requirements. Can be expanded in a future phase by adding new values to the `AuditAction` enum.

### `onDelete: Restrict` on `userId`
Block user deletion while audit logs reference them. Rejected: it would prevent deactivating staff accounts that have any audit history, which could become a permanent blocker as audit entries accumulate.

### `onDelete: Cascade` on `userId`
Delete audit log entries when the user is deleted. Rejected: this defeats the purpose of an audit log — the record of what happened must survive the deletion of the actor.
