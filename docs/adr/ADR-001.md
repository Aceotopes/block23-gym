# ADR-001 — Soft Delete on Client and User

**Status:** Accepted
**Date:** 2026-06-20
**Phase:** Phase 1 — Schema Redesign

---

## Context

The current `deleteClient()` server action executes a hard cascade delete, permanently removing the client record and all related attendance, payment, and membership records. This destroys revenue history and attendance data irreversibly.

Additionally, a staff management UI is planned for a future phase. When staff accounts are deactivated, their historical actions (audit log entries) must remain traceable.

Two models require a soft delete strategy: `Client` and `User`.

---

## Decision

Add a `deletedAt DateTime?` field to both `Client` and `User`.

- `null` means the record is active
- A non-null timestamp means the record is soft-deleted, with the value recording when the deletion occurred

### Client

- `deleteClient()` sets `deletedAt = now()` instead of calling `prisma.client.delete()`
- No cascade to related records — `Attendance`, `Payment`, and `Membership` records are preserved intact
- All active-client queries must include `where: { deletedAt: null }`
- An `AuditLog` entry with `action: DELETE_CLIENT` is written at the same time

### User

- `User.deletedAt` is added in Phase 1 even though the staff management UI is deferred
- Adding it now avoids a separate migration when the UI is eventually built
- All staff account queries must filter `where: { deletedAt: null }`
- A soft-deleted user can no longer log in (NextAuth credential check must verify `deletedAt IS NULL`)

---

## Consequences

**Positive:**
- Client deletion history (attendance, payments, memberships) is permanently preserved
- Revenue reports remain accurate even after a client is "removed" from the active list
- Soft-deleted clients can be surfaced in a future "archived clients" view
- User soft delete prevents broken audit log references

**Negative:**
- Every query that lists active clients must remember to add `where: { deletedAt: null }`. A missed filter will silently surface deleted clients as if they were still active.
- Database table size grows over time since records are never physically removed. Not a concern at the projected scale (under 500 clients in year 1).

**Scope of filter changes required in Phase 1:**
- `src/app/(dashboard)/clients/page.tsx` — main client list query
- `src/lib/client-status.ts` — `getClientType()` and `getClientStatus()` functions
- Any future dashboard query that counts "active clients"

---

## Alternatives Considered

### Hard delete with archiving to a separate table
Copy the client record to a `DeletedClient` archive table before deleting. Rejected: adds complexity, requires schema synchronization between two tables.

### Status enum on Client (ACTIVE / ARCHIVED)
Replace soft delete with a status field. Rejected: a `deletedAt` timestamp is strictly more informative — it records *when* the deletion occurred and enables time-based queries (e.g., "clients deleted in the last 30 days"). A status field does not.

### Cascade soft delete to related records
When a client is soft-deleted, also soft-delete their attendance, payment, and membership records. Rejected: the purpose of preserving history is undermined if related records are also hidden. Attendance and revenue reports must include data from deleted clients.
