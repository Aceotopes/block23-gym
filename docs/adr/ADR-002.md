## Context

Membership plans (1 Month / 2 Months / 3 Months at â‚±1,200/month) are currently hardcoded in three separate dialog components. Changing a fee or adding a plan requires editing source code and redeploying. The developer confirmed that admin-configurable plans are a requirement.

Two structural questions arise:

1. How should the `Membership` model reference a plan?
2. What happens when a plan is archived?

---

## Decision

### New `MembershipPlan` model

Admin-managed table with the following fields:
- `name String` â€” display name (e.g., "3 Months")
- `durationInDays Int` â€” duration in days
- `price Decimal(10,2)` â€” current price; â‚±0.00 is valid (free/promotional plans)
- `isActive Boolean @default(true)` â€” false means archived
- `sortOrder Int @default(0)` â€” controls display order in plan selector dropdowns

### Nullable `Membership.planId`

A nullable FK from `Membership` to `MembershipPlan`:
- Nullable in the schema to avoid breaking pre-migration membership records (which have no plan reference)
- Required in application code â€” the server action enforces that `planId` must be provided for all new memberships
- `durationInDays` is retained on `Membership` as a denormalized copy of the plan's duration at the time of purchase, since the plan's duration could be edited after the membership was created

### Referential action: `Restrict`

A `MembershipPlan` record cannot be deleted while any `Membership` references it. Admins retire plans by setting `isActive = false` (archiving), not by deleting them.

---

## Consequences

**Positive:**
- Admin can create, edit, and archive plans without a code deployment
- Plan-based reporting (e.g., "how many 3-month plans sold this month") becomes a simple filter query
- `sortOrder` prevents display-order instability as plans are created and archived over time
- â‚±0.00 plans enable promotional and trial memberships without special casing

**Negative:**
- `Membership` retains both `planId` and `durationInDays`. The duplication is intentional but must be documented â€” `durationInDays` is the authoritative value for membership length calculations; `planId` is for categorization and reporting.
- After archiving a plan, it no longer appears in the plan selector but its name is still visible on historical membership records via the FK join. This is the correct behavior.

**Query implication:**
- When displaying active plan options in the membership creation form, always filter `where: { isActive: true }` ordered by `sortOrder ASC`
- When joining for historical reporting, do not filter by `isActive` â€” archived plan names must appear on historical records

---

## Alternatives Considered

### Constants file instead of database table
Store plans in a TypeScript constants file. Rejected: the developer explicitly confirmed plans must be configurable by admin at runtime without a code deployment.

### Embed plan name on Membership (denormalize fully)
Copy `planName` onto `Membership` at creation time; no FK. Rejected: the plan name becomes read-only on the membership record â€” a name correction on the plan would not retroactively update it. A FK join gives the current canonical name while `durationInDays` and `amountPaid` preserve the exact terms of the original purchase.

### Cascade delete plan to memberships
If a plan is deleted, set `Membership.planId = null`. Rejected in favor of Restrict: deletion of a plan that has historical memberships should be prevented, not silently carried out. Archiving is the intended retirement path.
